## 1. 业务背景与对接目标

### 1.1 业务背景

车辆管理系统已完成车辆资产台账、调度管理等静态管理功能的建设，进入中期建设阶段。中期目标要求"基于已有的车辆静态数据，围绕车辆运行过程中产生的费用数据、车务数据等进行数字化管理，实现与售后系统等打通"。

售后管理系统负责车辆维修工单和保养工单的执行管理，是车辆运行阶段的核心业务系统。两个系统之间存在天然的数据依赖关系：车辆管理系统掌握车辆档案和质保信息，售后管理系统掌握工单执行和零部件更换数据，需要双向打通以形成车辆全生命周期数据闭环。

### 1.2 对接目标

| 目标维度    | 说明                                          |
| ------- | ------------------------------------------- |
| 工单结果可追溯 | 维修/保养工单执行结果自动同步至车辆管理系统，形成完整维保档案            |
| 零部件变更可追踪 | 零部件更换记录自动同步，支持物料级追溯（自研车辆关联BOM，其他车辆关联维修记录）  |
| 质保判断可联动 | 车辆管理系统质保信息推送给售后系统，支撑"是否质保内"的精确判断          |
| 车辆状态可同步 | 工单生命周期驱动车辆状态变更，避免两系统状态不一致                  |
| 维保计划可自动更新 | 保养工单完成后自动更新维保计划，避免人工重复录入                   |

### 1.3 使用角色

| 角色     | 在对接场景中的职责                     |
| ------ | ----------------------------- |
| 管理员    | 查看所有对接数据、处理同步异常、手动刷新数据       |
| 项目经理   | 查看权限范围内车辆的对接数据               |
| 财务人员   | 查看对接数据（质保状态影响资产价值判断）        |
| 售后系统   | 推送工单结果和零部件变更，接收质保和车辆信息      |
| 车辆管理系统 | 接收工单数据，推送质保/车辆/维保计划信息       |

## 2. 系统交互架构

### 2.1 整体数据流向

```
┌──────────────────┐     推送（售后系统主动）      ┌──────────────────┐
│                  │ ──────────────────────────→ │                  │
│   售后管理系统     │   维修工单执行结果              │   车辆管理系统     │
│                  │   保养工单执行结果              │                  │
│                  │   零部件更换记录                │                  │
│                  │   车辆状态变更（维修中/保养中）    │                  │
│                  │                              │                  │
│                  │ ←────────────────────────── │                  │
│                  │     推送（车辆管理系统主动）      │                  │
│                  │   车辆基础信息变更              │                  │
│                  │   质保信息变更                  │                  │
│                  │   维保计划变更                  │                  │
│                  │   车辆状态变更（报废等）          │                  │
│                  │   质保到期提醒                  │                  │
└──────────────────┘                              └──────────────────┘

                    ┌──────────────────┐
                    │   第三方MES系统     │ ← BOM清单数据来源，双方均从MES获取
                    └──────────────────┘
```

### 2.2 推送模式

采用**主动推送为主 + 手动刷新为辅**的混合模式：

| 模式     | 说明                                          | 适用场景          |
| ------ | ------------------------------------------- | ------------- |
| 主动推送   | 数据变更后实时推送，确保时效性                             | 常规数据同步        |
| 主动查询   | 售后系统在创建工单前可主动查询车辆管理系统获取最新质保/车辆信息            | 工单创建前确认最新状态   |
| 手动刷新   | 车辆管理系统提供"手动刷新"按钮，用户可主动触发重新拉取                | 推送失败或网络异常时兜底  |

## 3. 对接内容详细定义

### 3.1 售后系统 → 车辆管理系统

#### 3.1.1 维修工单执行结果推送

**触发时机：** 维修工单完成或取消时，由售后系统主动推送。

**推送业务内容：**

| 业务信息       | 字段标识                  | 说明                                       |
| ---------- | --------------------- | ---------------------------------------- |
| 车架号        | vin_code              | 关联车辆主键                                   |
| 工单号        | work_order_no         | 售后系统工单编号                                 |
| 工单类型       | work_order_type       | 固定为"repair"（维修）                          |
| 工单状态       | work_order_status     | completed-完成 / cancelled-取消              |
| 维修时间       | repair_time           | 格式：YYYY-MM-DD HH:mm:ss                   |
| 维修时里程      | repair_mileage        | 单位：公里                                    |
| 维修项目       | repair_items          | 维修的具体项目描述                                |
| 是否质保内      | is_in_warranty        | 由售后系统判断（基于车辆管理系统推送的质保信息）                 |
| 维修费用       | repair_cost           | 单位：元；质保内维修费用为0                           |
| 维修服务商      | repair_provider       | 本次维修的服务商名称                               |
| 维修说明       | repair_remark         | 维修备注说明                                   |
| 车辆状态变更-变更前 | vehicle_status_before | 工单创建前的车辆状态                               |
| 车辆状态变更-变更后 | vehicle_status_after  | 工单完成/取消后的车辆状态                            |
| 零部件变更明细    | parts_changes         | 零部件变更列表（可能为空），结构见3.1.3节                  |

**车辆状态变更规则：**

| 工单事件     | 推送的车辆状态变更                     |
| -------- | ----------------------------- |
| 工单创建     | 变更前=当前状态，变更后="在库-维修"          |
| 工单完成     | 变更前="在库-维修"，变更后=工单创建前状态       |
| 工单取消     | 变更前="在库-维修"，变更后=工单创建前状态（回退）   |

**车辆管理系统接收后的处理逻辑：**

1. 写入维修记录表（vehicle_repair_record）
2. 更新车辆状态字段（vehicle_status）
3. 写入零部件变更记录（vehicle_bom_change），仅自研车辆同时更新BOM清单Tab数据
4. 自动记录变更日志（操作来源标注"售后系统"）

---

#### 3.1.2 保养工单执行结果推送

**触发时机：** 保养工单完成或取消时，由售后系统主动推送。

**推送业务内容：**

| 业务信息       | 字段标识                    | 说明                                       |
| ---------- | ----------------------- | ---------------------------------------- |
| 车架号        | vin_code                | 关联车辆主键                                   |
| 工单号        | work_order_no           | 售后系统工单编号                                 |
| 工单类型       | work_order_type         | 固定为"maintenance"（保养）                     |
| 工单状态       | work_order_status       | completed-完成 / cancelled-取消              |
| 保养时间       | maintenance_time        | 格式：YYYY-MM-DD HH:mm:ss                   |
| 保养时里程      | maintenance_mileage     | 单位：公里                                    |
| 保养项目       | maintenance_items       | 保养的具体项目描述                                |
| 是否质保内      | is_in_warranty          | 保养也可能在质保期内（如免费首保），由售后系统判断               |
| 保养费用       | maintenance_cost        | 单位：元                                    |
| 保养服务商      | maintenance_provider    | 本次保养的服务商名称                               |
| 保养说明       | maintenance_remark      | 保养备注说明                                   |
| 车辆状态变更-变更前 | vehicle_status_before   | 工单创建前的车辆状态                               |
| 车辆状态变更-变更后 | vehicle_status_after    | 工单完成/取消后的车辆状态                            |
| 零部件变更明细    | parts_changes           | 保养中涉及的零部件更换列表（通常为空，保养时更换零件则填写），结构见3.1.3节 |

**车辆状态变更规则：** 与维修工单一致，状态为"在库-保养中"。

**车辆管理系统接收后的处理逻辑：**

1. 写入保养记录表（vehicle_maintenance_record）
2. 更新车辆状态字段（vehicle_status）
3. 写入零部件变更记录（vehicle_bom_change），仅自研车辆同时更新BOM清单Tab数据
4. **自动更新维保计划**（详见3.1.2.1节）
5. 自动记录变更日志（操作来源标注"售后系统"）

##### 3.1.2.1 保养工单完成后维保计划自动更新规则

| 自动行为             | 触发条件                    | 处理逻辑                                       |
| ---------------- | ----------------------- | ------------------------------------------ |
| 自动填充首保日期        | 首保日期（first_maintenance_date）为空 | 将保养时间写入首保日期                                |
| 自动填充首保里程        | 首保里程（first_maintenance_mileage）为空 | 将保养时里程写入首保里程                               |
| 自动重算已保养次数       | 始终执行                    | 从保养记录列表中统计该车辆的保养记录条数（实时计算，不存储）             |
| 自动重算下次保养提醒日期    | 首保日期和保养周期均已填写           | 按PRD 3.3.8节计算规则重算                          |
| 自动更新概览统计条质保状态   | 保养时里程可能影响质保状态           | 里程变化后重新判断质保状态（里程条件）                        |

**变更日志记录：** 上述自动更新均在变更日志中记录，操作类型为"维保计划自动更新"，操作来源为"售后系统"。

#### 3.1.3 零部件变更明细

零部件变更明细嵌套在维修/保养工单推送中，不单独推送。

**变更明细结构：**

| 业务信息    | 字段标识             | 说明                                       |
| ------- | ---------------- | ---------------------------------------- |
| 件号      | part_no          | 零部件编号                                    |
| 名字      | part_name        | 零部件名称                                    |
| 变更类型    | change_type      | replace-更换 / add-新增 / remove-移除           |
| 变更前件号   | before_part_no   | 变更前的件号；新增类型为空                            |
| 变更前供应商  | before_supplier  | 变更前的供应商                                  |
| 变更前规格型号 | before_spec      | 变更前的规格型号                                 |
| 变更后件号   | after_part_no    | 变更后的件号；移除类型为空                            |
| 变更后供应商  | after_supplier   | 变更后的供应商                                  |
| 变更后规格型号 | after_spec       | 变更后的规格型号                                 |
| 变更原因    | change_reason    | 更换/新增/移除的原因描述                            |

**零部件变更数据的展示策略：**

| 车辆资产来源 | 展示位置                                         |
| ------ | -------------------------------------------- |
| 自研     | BOM清单Tab → 变更记录视图（已有设计）+ 当前BOM视图自动合并更新      |
| 非自研    | 售后维保Tab → 维修信息/保养信息子Tab → 记录行"涉及零部件"列展开查看明细 |

## 4. 对接后页面变更说明

### 4.1 售后维保Tab — 维修信息子Tab（激活）

对接后，维修信息子Tab从"暂未开放"状态激活为正式功能。

#### 4.1.1 列表字段

| 序号 | 字段名      | 字段标识               | 宽度建议   | 对齐方式 | 说明                                           |
| -- | -------- | ------------------ | ------ | ---- | -------------------------------------------- |
| 0  | 复选框      | checkbox           | 50px   | 居中   | 支持全选/单选，用于导出操作                               |
| 1  | 维修时间     | repair_time        | 180px  | 左对齐  | 格式：YYYY-MM-DD HH:mm:ss                       |

## 5. 异常场景与处理规则

### 5.1 推送失败处理

| 异常场景     | 处理规则                                          |
| -------- | --------------------------------------------- |
| 推送失败     | 推送方自动重试3次，间隔递增（5分钟/15分钟/30分钟）                 |
| 3次重试仍失败  | 标记为"推送异常"，需人工干预                               |
| 接收方兜底    | 车辆管理系统提供"手动刷新"按钮，用户可主动触发重新拉取                  |

### 5.2 重复推送处理（幂等性）

| 规则项   | 说明                          |
| ----- | --------------------------- |
| 幂等依据  | 以**工单号**为幂等键               |
| 重复推送  | 同一工单号多次推送，以最新一次为准，覆盖更新     |

### 5.3 乱序推送处理

| 规则项   | 说明                                    |
| ----- | ------------------------------------- |
| 排序依据  | 以**工单完成时间**为准排序展示，不受推送到达顺序影响         |
| 状态判断  | 车辆状态以最新推送的状态为准                       |

### 5.4 部分成功处理

| 规则项   | 说明                                    |
| ----- | ------------------------------------- |
| 处理策略  | **整体回滚**，工单信息和零部件变更必须同时写入成功          |
| 理由    | 避免"有维修记录但没有零部件变更"的数据不一致问题            |

### 5.5 并发编辑处理

| 场景                     | 处理规则                          |
| ---------------------- | ----------------------------- |
| 车辆维修中，管理员编辑车辆基础信息     | 允许编辑，变更正常推送给售后系统              |
| 车辆维修中，管理员编辑质保信息       | 允许编辑，变更正常推送给售后系统              |
| 质保信息变更对当前工单的影响        | 已创建的工单以**创建时**的质保状态为准，后续变更不影响 |

### 5.6 售后系统创建工单前的车辆状态校验

| 规则项   | 说明                                            |
| ----- | --------------------------------------------- |
| 校验方式  | 售后系统创建工单前查询车辆管理系统获取当前车辆状态                     |
| 校验结果  | 若车辆已在"维修中/保养中"，**仅做提示**，不阻止创建工单               |
| 理由    | 提示但不阻止，由售后系统操作人员自行判断（可能存在紧急维修场景）              |

### 5.7 工单号跳转异常处理

| 异常场景         | 处理规则                          |
| ------------ | ----------------------------- |
| 售后系统对应工单已删除 | 显示"该工单已被删除或不存在"提示页           |
| 售后系统不可访问    | 显示"售后系统暂时无法访问，请稍后重试"提示       |
| SSO免登失败     | 跳转至售后系统登录页，由用户手动登录           |

## 6. 历史数据初始化

### 6.1 历史数据同步

对接上线时，需将售后系统中已有的历史工单数据一次性同步至车辆管理系统。

| 规则项     | 说明                          |
| ------- | --------------------------- |
| 同步范围    | 近2年内的维修/保养工单记录              |
| 同步方式    | 一次性批量同步                     |
| 数据来源标注  | 历史数据在记录中标注数据来源为"历史导入"       |

### 6.2 维保计划回算

历史数据同步完成后，自动执行维保计划回算：

| 回算项       | 说明                              |
| --------- | ------------------------------- |
| 首保日期/里程   | 从保养记录中取最早一条，回填首保日期和首保里程        |
| 已保养次数     | 从保养记录中统计条数                     |
| 下次保养提醒日期 | 基于已保养次数 + 保养周期重算               |
| 质保状态      | 基于最新里程数据重新判断                   |
